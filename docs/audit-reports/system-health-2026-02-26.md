## Стек и архитектура

```
React 18 (CRA)
  → Netlify Edge Function /api/generate-lesson  [боевой путь: SSE-стриминг]
  → Claude Haiku 4.5 API
  → Netlify Regular Functions: save-lesson, get-analytics, list-pending-users,
                               update-user-status, notify-admin
  → GitHub API (хранилище уроков)
  → Supabase Auth (аутентификация + аналитика)
  → Resend (email уведомления)
```

## Блок 1 — Архитектура

**Оценка: 3/10**

Разделение Edge/Regular функций по назначению разумное: Edge для SSE-стриминга без таймаута CDN, Regular для остального. Supabase Auth и GitHub как хранилище — элегантное решение для команды без собственного бэкенда.

**Критические находки:**

- `src/App.jsx` — 2104 строки в одном файле. Содержит UI, бизнес-логику, AI pipeline, библиотеку и экспорт. God component в полном смысле.
- `netlify/functions/generate-lesson.js` — **мёртвый код**. Функция написана, задеплоена, но никогда не вызывается в боевом пути: клиент ходит на Edge endpoint `/api/generate-lesson`, а не на regular function.
- React CRA — инструмент officially deprecated. Для новых задач это технический долг с первого дня.
- CORS-блоки и admin JWT-проверка дублируются в трёх функциях без общего middleware.
- Контур хранения связан с контуром деплоя: запись уроков в `public/lessons/` превращает контент-операцию в CI/CD-операцию (структурная связность данных и релизного пайплайна).
- Проектная структура монолитная: в `App.jsx` одновременно сосуществуют prompt-engineering, сетевые вызовы, parsing/repair, рендер, экспорт `.doc`, библиотека и рефлексия (низкая модульность, высокая стоимость изменений).

## Блок 2 — Data Flow

**Оценка: 3/10**

Цепочка генерации прозрачная. Curriculum JSON загружается через `fetch` с кешем в `useRef`. System prompt строится в `buildSystemPrompt()` из 6 параметров. SSE-стриминг обрабатывается клиентом через ReadableStream.

**Критические находки:**

- **Аналитика генераций мертва.** Событие `generated` пишется в regular `generate-lesson.js` (строки 54–134), но боевой путь идёт через Edge Function, которая не содержит логирования. Итог: таблица `lesson_events` с `event_type: generated` в Supabase пуста или содержит данные только из тестовых прямых вызовов regular function.
- Реальная статистика активности = только 6 записей в `public/lessons/index.json` (сохранения). Все управленческие решения по активности принимаются на неполных данных.
- `extractAndRepair` — при обрыве генерации внутри строкового значения JSON (незакрытая кавычка) добавляет `}}` внутрь строки. Результат: валидный JSON с семантически сломанной структурой. Пользователь получает урок с обрезанным текстом без сообщения об ошибке — хуже, чем явный crash.
- Нет серверной схемной валидации входа в `save-lesson` (валидируется только наличие части полей), что допускает структурно неконсистентные payload.
- Обновление `public/lessons/index.json` уязвимо к lost update при параллельных сохранениях (read-modify-write без защиты от гонки/версий).
- Идентификатор урока `subject-grade-lesson_num` может коллидировать и скрывать часть уроков в индексе при одинаковом `lesson_num`.

## Блок 3 — Безопасность

**Оценка: 1/10**

**Критические находки:**

**[CRITICAL]** `save-lesson.js` не проверяет JWT и роль пользователя вообще (строки 17–213). Любой POST-запрос на `/.netlify/functions/save-lesson` с валидным JSON записывает файлы в GitHub через серверный `GITHUB_TOKEN`. Плюс: `public/lessons/` деплоится как статика — каждый коммит через GitHub API триггерит полный Netlify deploy. Это не просто "замусоривание репо" — это возможность вызывать CI/CD цикл на чужие вычислительные ресурсы через публичный endpoint.

**[HIGH]** `generate-lesson` (Edge): нет rate limiting. Нет whitelist моделей — клиент передаёт `model` и `max_tokens` произвольными, Edge проксирует их как есть в Anthropic API. Любой авторизованный пользователь может передать `max_tokens: 100000` или подставить более дорогую модель.

**[CRITICAL]** `generate-lesson` (Edge): на endpoint не реализована серверная проверка JWT/роли. В текущем виде это публичный прокси к Anthropic API.

**[HIGH]** CORS: `Access-Control-Allow-Origin: "*"` во всех функциях включая те, что работают с Supabase Service Role Key.

**Приемлемо:**
- `ANTHROPIC_API_KEY`, `GITHUB_TOKEN`, `SUPABASE_SERVICE_ROLE_KEY` — только в Netlify env vars, не в репо.
- Admin-check в `get-analytics` и `list-pending-users` корректен: JWT → `/auth/v1/user` → `role === admin`.
- Supabase anon key в `netlify.toml` — публичный клиентский ключ, допустимо.

## Блок 4 — AI / Промпт-инжиниринг

**Оценка: 3/10**

`buildSystemPrompt()` — наиболее продуманная часть системы. Дифференцированные промпты по возрастным группам (isPrimary/isMiddle/старшая), предметным кластерам, форматам. Разные JSON-схемы для начальной и средней школы. Контекст учебной программы (предыдущие/следующие уроки). Персонаж Кори с конкретными ролями.

**Технические находки:**

- Рассинхрон model ID: клиент отправляет `claude-haiku-4-5-20251001`, Edge default: `claude-3-5-haiku-20241022`. В боевом пути используется клиентский ID (передаётся явно) — но default в Edge потенциально опасен при прямых вызовах.
- Рассинхрон `max_tokens`: клиент 8000, Edge default 2000. В боевом пути 8000 (override). Edge default не имеет значения, но создаёт ложное ощущение ограничения.
- `repairJsonStrings` + `extractAndRepair`: логика необходима и нетривиальна, но подсчёт `{}/[]` не учитывает контекст строк. Ненадёжна на краевых случаях.

## Блок 5 — Учебный контент

**Оценка: 4/10**

33 curriculum файла в `public/curriculum/`. Покрытие широкое для российской школьной программы.

**Пробелы:**

- Без curriculum полностью: Английский язык, Немецкий язык, Французский язык, МХК. Предметы видны в UI, пользователь их выбирает, но программный контекст не передаётся в промпт.

- Смешанные структуры JSON: часть файлов lesson-level (подробные), большинство — sections-only (только разделы без отдельных уроков).

## Блок 6 — Операционная зрелость

**Оценка: 2/10**

CI/CD через Netlify auto-deploy настроен. Git история чистая с осмысленными commit messages.

**Находки:**

- Нет тестов. `npm test` в `package.json` есть, тест-файлов нет ни одного.
- Нет error tracking (Sentry, Datadog или аналог).
- Нет GitHub Actions. CI только через Netlify build hook.
- Git история засорена автокоммитами данных (`Урок: ...`, `Индекс: ...`) — они мешают читать историю изменений кода.
- Таймауты функций заданы корректно (10/15/30 сек). Edge без таймаута.
- Fallback при недоступном Supabase: генерация (Edge) не зависит от Supabase — OK. Логирование в save-lesson best-effort — не валит сохранение — OK.
- Отсутствуют технические guardrails для бюджетов LLM (quota control, per-user throttling, model allowlist на сервере).
- Нет проверяемого CI-quality gate перед merge в `main` (структурный риск регрессий в проде).

## Топ-3 приоритета для следующего спринта

1. **Закрыть `save-lesson`** — добавить JWT-проверку + проверку статуса пользователя. Рассмотреть отвязку от auto-deploy триггера (перенос в отдельное хранилище или S3).
2. **Закрыть Edge endpoint** — серверная JWT-проверка + rate limit + whitelist моделей + ceiling `max_tokens`.
3. **Стабилизировать поток сохранения** — устранить коллизии `id`, защитить обновление `index.json` от гонок, выровнять аналитику `generated` в боевом пути.
