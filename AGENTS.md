## 3. Документация проекта

### Расположение

Все документы проекта хранятся в:
```
/Users/popskraft/Projects/koriphey-org/01-zhivoi-urok-constructor/docs/
```

> Это абсолютный путь на машине владельца. В рабочей сессии AI доступ через примонтированную папку.

### ⚠️ Исключение из глобальных правил экосистемы

В глобальных правилах Koriphey системная документация хранится в `_docs/` (с underscore-префиксом).
**В этом проекте это правило не применяется.**

Папка документации: `docs/` — без underscore-префикса.
Причина: проект существовал до введения стандарта `_docs/`, папка `docs/` уже использовалась и содержит связанные файлы. Переименование нарушит существующие ссылки.

AI-агент обязан использовать `docs/` — и только её. Создание `_docs/` в этом проекте запрещено.

### Структура

```
docs/
├── project-rules.md              ← правила проекта (100% приоритет)
├── roadmap-goals.md              ← дорожная карта, цели, вехи
├── development-log.md            ← канонический devlog (авто, версионированный)
├── github-netlify-setup.md       ← общая документация
├── audit-reports/                ← отчёты о состоянии системы
│   └── system-health-YYYY-MM-DD.md
└── specs/                        ← технические спецификации фич
    └── lesson-library.md
```

- В корне `docs/` — системные и общие документы проекта.
- Специфические документы (отчёты, исследования, аудиты) — в подпапках, создаваемых автоматически по контексту с именами в kebab-case.

## 5. Обязательные действия AI-агента

### При каждой рабочей сессии

1. Прочитать `AGENTS.md` — этот файл
2. Прочитать `docs/project-rules.md` — если существует
3. Прочитать хвост `docs/development-log.md` — понять последний контекст (последняя версия и последние записи)
4. Прочитать `docs/roadmap-goals.md` — понять текущие приоритеты

### После завершения работы

- Записать в `docs/development-log.md` всё что сделано (формат ниже)
- Обновить статусы задач в `docs/roadmap-goals.md` если что-то выполнено или изменилось

### При критических изменениях

Критическими считаются: изменение архитектуры, новые фичи, security-фиксы, деплой, изменение правил.
При каждом таком событии обязательно обновить `roadmap-goals.md` и `development-log.md`.

## 6. Канонический лог разработки — development-log.md

**`activity-log.md` упразднён.** Единственный лог проекта — `docs/development-log.md`.

### Структура файла

Файл разделён на версии (`## v1`, `## v2`, ...). Каждая версия — логический этап разработки.
Внутри версии — записи по датам (`### YYYY-MM-DD — Описание дня`).

### Формат записи

```
- [x] id:YYYYMMDD-HHMMSS | тип | что сделано | email@domain.com | СТАТУС
```

Типы: `setup`, `feature`, `fix`, `docs`, `audit`, `security`, `chore`
Статусы: `DONE`, `REVIEW`, `REJECTED`

ID = timestamp момента записи. Числовые порядковые ID запрещены.

### Правила ведения

- Новые записи добавлять СНИЗУ в текущую версию
- При старте нового этапа разработки — добавить секцию `## vN` с датой и кратким описанием
- Старые записи не удалять никогда
- При старте новой версии НЕ создавать новый файл — всё в одном `development-log.md`

## 7. Правила поведения агента

- Перед изменением файлов — сначала прочитать контекст
- Не изменять `AGENTS.md` без явного запроса владельца
- Не удалять файлы — только перемещать в архив или менять `status: archive`
- Сложные задачи — итерациями: изменил → проверь → следующий шаг
- Не переделывать работающее без явного запроса
- При работе с кодом: только читать и анализировать, если иное не указано явно
